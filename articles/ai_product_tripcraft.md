最後の更新から気づけばだいぶ時間が空いてしまいました。
サボっていた理由は単純で、AI開発が楽しすぎたからです。

Claude Codeを使ってWebアプリを作り始めたら、これが想像以上に面白くて、記事を書く暇があったらコード書いてました。
マンガ読んだりゲームする時間も、全部AIに吸われましたね。


## 作ったもの：tripcraft.fun

旅行の計画をみんなで楽しく作るWebアプリを作りました。

https://tripcraft.fun

![tripcraft.funのトップページ](/images/ai_product_tripcraft/top.webp)

「やりたいこと」を出して、それを旅程に組み立てていく、というコンセプトです。
ありがちな「決まった予定を入力するだけ」のアプリではなく「計画を立てる過程そのものを楽しむ」ためのアプリです。


### 使い方

1. 行先・日程などの基本情報で旅のプランを作成
2. 招待リンクでメンバーを招待
3. 「やりたいこと」を追加して、メンバーとリアクションやコメントで盛り上がる
4. やりたいことを盛り込んだ旅程を作成

![やりたいことの一覧](/images/ai_product_tripcraft/wish_items.webp)

### その他特徴など
- ログインなしでも使える（データはlocalStorageに保存）
- ログインすればメンバーを招待できる、ゲスト参加も可能
- オフラインも行けるはず（テスト甘いかも）
- 将来的にはAI機能も追加します

![旅程](/images/ai_product_tripcraft/itinerary.webp)



### 技術スタック

- **フロントエンド**: React + TypeScript + Tailwind CSS
- **ルーティング**: TanStack Router（ファイルベース）
- **バックエンド**: Firebase Functions
- **DB**: Cloud Firestore
- **認証**: Firebase Authentication
- **ホスティング**: Firebase Hosting

いわゆるFirebaseにフルベットした構成です。ログインなしでも使えるようにlocalStorageとFirestoreのデュアルライト方式にしていたり、招待リンクでのゲスト参加にカスタムトークン認証を使っていたり、まぁまぁ凝った設計になってます。


## Claude Codeでの開発スタイル


### vibeコーディング？

いわゆる「vibeコーディング」に含まれる開発スタイルだと思うのですが、当初思ってたより細かく指示を出しましたね。後ろに数字を載せます。
最終的なAI開発の流れは以下のスタイルに落ち着いていて、１バックログごとに以下を行う感じです。

1. 設計、方針、データ構造について指示してAIに提案させる
2. 納得いくまでラリーを繰り返して決定する
3. ドキュメントやデータ構造のみを実装させる
4. 処理の実装計画を提案させる
5. 納得いくまでラリーを繰り返して決定する
6. 処理を実装させる
7. 出来上がったものの動作確認
8. 触って気になったコトを修正or別バックログに積み増し
9. 完了

これって、よくあるアジャイル開発の姿だなーって印象です。ただ、スピード感が１０倍ぐらいなのですけどね。よく言われることですが、AIは開発を加速させる。をまさに実感しましたね。


### 数字で見る開発

約7週間（主に土日＋平日夜）の開発で、こんな数字になりました。

| 項目                     | 数値                 |
|--------------------------|----------------------|
| 総コミット数             | 約800回              |
| AI開発コミット           | 約450回（全体の56%） |
| バックログ管理コミット   | 約100回（全体の13%） |
| ドキュメント更新コミット | 約250回（全体の31%） |
| AIが書いたコードの割合   | 約98%（定性評価）    |

（バックログ管理やドキュメント更新を設計とみなすと、ウォーターフォールの開発工程比率に近い数字なんですよね）


### AIの実力

やってみた感覚として、AIの提案は2〜3割くらいの確率で保守性を下げる提案をしてきます。
見えているスコープが狭くて、保守性の勘所が若手技術者レベルって感覚がぴったりきます。

特に、ドキュメント、データ構造などはイマイチですね。
逆に、処理を書かせるときちんとしたコードが出てくることが多かったです。
なので、方針やデータ構造についてAIと細かくラリーして決めてから、開発をやらせるというスタイルに落ち着きました。


### やはりドキュメントが重要

CLAUDE.mdに加えて、10ファイル程度のドキュメントを整備しました。アーキテクチャー、データモデル、機能仕様、デザインガイドライン、テスト方針…。これらをAIが参照することで、プロジェクト全体の方針と一貫した実装が出てくる確率がかなり上がりました。

また、実装しているとドキュメントと実装の乖離が出てくるので、ドキュメントもちゃんとリファクタリングしていく必要があります。


### バックログ管理は人間がやった方が良いかもしれない

バックログ管理のコミットが100回というのは、自分でも多いなと思います。
でも、並列作業させるし、スピードが速するので、すぐに状況を見失ってしまうんですよね。なので、自分のためにもバックログはAIに任せず手動で更新していました。（内容は考えてもらったりしてましたが）


## まとめ

Claude Codeを使った開発は本当に楽しいですね。当面飽きずに続けられそうです。でも「AIに任せれば全部やってくれる」わけではなくて、むしろ設計とタスク管理を人間がしっかりやることで、AIの力を引き出せるという実感です。

tripcraft.funはまだまだ発展途上で、AI機能の搭載など、やりたいことがたくさんあります。
また更新が止まったら、きっと何か作ってるんだと思ってください。
